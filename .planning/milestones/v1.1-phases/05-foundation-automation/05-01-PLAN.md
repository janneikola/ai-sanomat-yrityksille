**3. Schedule Service (api/src/services/scheduleService.ts — NEW):**

Create with these exports:
- `isDueToday(frequency, preferredDay, biweeklyWeek, today?)` — pure function, checks if client is due today
- `getPeriodNumber(frequency, date?)` — returns period number for dedup. Weekly: ISO week (1-53). Biweekly: `Math.min(Math.ceil(isoWeek / 2), 26)`. Monthly: month 1-12.
- `getISOWeekNumber(date)` — extracted from newsletterService.ts `getWeekNumber()` logic, takes a date param instead of using `new Date()`
- `getNextScheduledDate(frequency, preferredDay, biweeklyWeek, fromDate?)` — finds next occurrence after fromDate, max 35 day search loop
- `processScheduledDigests()` — main scheduler entry point:
  1. Query active clients where `schedulePaused = false`
  2. Filter clients where `isDueToday()` returns true
  3. For each client (sequentially, NOT Promise.all — respects API rate limits):
     a. Calculate `periodNumber` and `year`
     b. Check if issue already exists for this `clientId + periodNumber + year` — skip if exists (SCHED-04)
     c. Determine news window (`sinceDate`): find most recent `sent` or `approved` issue for client, use its `createdAt` as `sinceDate`. First digest (no prior issue): `sinceDate = new Date(Date.now() - 7 * 24 * 60 * 60 * 1000)` (last 7 days). If no qualifying issue exists, use undefined for first digest fallback.
     d. Count available news items in window (items with `collectedAt > sinceDate`). If < 5: email admin "below threshold, will retry tomorrow", record as skip, continue
     e. Call `generateClientDigest(clientId, sinceDate)` — pass the computed `sinceDate` so the generator filters news to the correct window
     f. On success: email admin with "draft ready" notification including link to preview page
     g. On failure: existing `generateClientDigest` already sets issue to 'failed'; additionally email admin about the failure
  4. Log scheduler run to `schedulerRuns` table
  5. Return summary: { processed, successes, failures, skips }
- `checkAndRunMissedSchedule()` — called on startup after 30s delay:
  1. Get today at midnight
  2. Query latest `schedulerRuns` entry
  3. If no run exists for today, call `processScheduledDigests()`
- `logSchedulerRun(data)` — inserts into `schedulerRuns` table

**4. Source Health Service (api/src/services/sourceHealthService.ts — NEW):**

Create with these exports:
- `computeHealthStatus(source)` — pure function returning 'green' | 'yellow' | 'red':
  - `!isActive` → 'red'
  - `consecutiveFailures >= 4` → 'red'
  - `consecutiveFailures >= 2` → 'yellow'
  - `lastItemsAt` exists AND `daysSince(lastItemsAt) >= 7` AND `lastItemCount === 0` → 'yellow' (stale)
  - Otherwise → 'green'
- `logFetchAttempt(sourceId, success, itemCount, errorMessage?)` — inserts into `sourceHealthLogs` table
- `updateSourceHealth(sourceId, success, itemCount)` — updates `newsSources` aggregate columns:
  - On success: reset `consecutiveFailures` to 0, update `lastSuccessAt`, update `lastItemCount`, update `lastItemsAt` if itemCount > 0
  - On failure: increment `consecutiveFailures`
- `checkAutoDisable(sourceId, consecutiveFailures)` — if >= 5: set `isActive = false`, send admin notification email
- `checkHealthTransitionNotification(sourceName, oldFailures, newFailures)` — send email when crossing yellow (2) or red (4) thresholds
